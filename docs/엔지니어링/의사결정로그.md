# 의사결정 로그 (ADR)

> **누가 봄**: FE / BE / 합류자
> **언제 봄**: "왜 이렇게 했지?" 의문 들 때, 새 결정 내릴 때

큰 결정만 시간순. 소소한 건 PR description에. 백엔드 결정은 [API 저장소 의사결정로그](https://github.com/giwon1130/baby-log-api/tree/main/docs/엔지니어링/의사결정로그.md).

## ADR-001 — Expo SDK + React Native (2025-Q4)

**문맥**: 신생아 부모 대상 모바일 앱. iOS 우선이지만 Android도 가능성 열어두기.

**대안**:
- 네이티브 Swift만 — 빠른 출시 어려움, Android 포기
- Flutter — Dart 신규 학습 필요, 네이티브 모듈 통합 RN보다 어려움
- Expo + RN — JS/TS 익숙, 양 플랫폼 단일 코드베이스, 빠른 반복

**결정**: **Expo SDK 54 + RN 0.81 + TypeScript**

**결과**: 좋음. 핫리로드로 빠른 반복. iOS 네이티브 모듈도 Expo 로컬 모듈 패턴으로 깔끔함.

## ADR-002 — Spring Boot + JdbcTemplate (백엔드, 2025-Q4)

자세한 건 [API 저장소 ADR-002](https://github.com/giwon1130/baby-log-api/tree/main/docs/엔지니어링/의사결정로그.md).

요약: JPA 안 쓰고 JdbcTemplate. 작은 도메인이라 JPA 추상화 비용 > 이득. SQL 명시적.

## ADR-003 — fly.io → Railway (2026-Q1)

**문맥**: 처음 fly.io 사용. PostgreSQL attach 후 secret 매핑 번거로움. Railway 가
PG plugin + 환경변수 자동 주입이 더 매끄러움.

**결정**: Railway로 이전. main 푸쉬 시 자동 배포.

**결과**: 배포 자체는 쉬워졌음. 무료 티어 cold start 2~3초는 감수. 비용은 [`운영/비용최적화.md`](../운영/비용최적화.md).

## ADR-004 — 울음 분석 온디바이스 처리 (2026-04)

**문맥**: ChatterBaby 등 경쟁 앱은 음성 파일을 서버 업로드. 데이터 비용 + 프라이버시
이슈. 부모가 아기 울음 녹음 보내는 거 거부감 큼.

**대안**:
- (A) 음성 파일 업로드 → 서버 ML 모델 추론
- (B) 모든 추론 온디바이스 (CoreML 모델 번들)
- (C) 온디바이스 feature 추출 + 서버는 휴리스틱/유사도 (선택)

**결정**: **(C)**. iOS Apple SoundAnalysis로 울음 감지 + 자체 Accelerate 기반
음향 feature 추출 → 숫자만 서버로. 음성 파일 일체 미저장.

**근거**:
- 프라이버시 우위 — 마케팅 포인트
- 서버 비용 ↓ (Railway free tier 유지 가능)
- 5초 녹음 후 즉시 결과 — UX 빠름
- 향후 진짜 ML 모델은 Phase 2B에서 YAMNet 임베딩 + 서버 k-NN 으로 확장

**트레이드오프**: 처음에는 정확도 낮음. 휴리스틱 + 학습 루프로 보완.

## ADR-005 — 휴리스틱 우선 → 학습 루프 (2026-04)

**문맥**: Donate-a-Cry 코퍼스 446개 샘플 중 71%가 "배고픔" → 학습용 데이터셋 한계.
처음부터 ML 모델 학습 불가.

**대안**:
- (A) 5-way CNN 학습 (실패 — 무조건 배고픔 예측)
- (B) 휴리스틱 (컨텍스트 + 음향 규칙) + per-baby 누적 학습 (similarity)

**결정**: **(B)**. 처음에는 컨텍스트 prior + 음향 규칙. 사용자 정정이 쌓이면
similarity로 가중. Phase 2B에서 YAMNet 임베딩 추가.

**핵심 임계값** (수정 시 [`제품/용어집.md`](../제품/용어집.md) 와 백엔드 코드 동기화):
- 피치 > 600Hz → PAIN +0.2
- 피치 < 250Hz → TIRED +0.12
- 리듬성 > 0.45 → HUNGER +0.15
- 마지막 수유 > 180분 → HUNGER +0.3

## ADR-006 — Expo 로컬 네이티브 모듈 (2026-04)

**문맥**: iOS SoundAnalysis + AVAudioEngine + Accelerate를 RN에서 호출 필요.

**대안**:
- (A) `react-native-audio-record` 같은 외부 패키지 + JS에서 모든 처리
- (B) 자체 네이티브 모듈 (`modules/cry-detector/`)

**결정**: **(B)**. 외부 라이브러리에 SoundAnalysis 통합 패키지 없음. 자체로 만드는 게
의존성 최소화에도 좋음.

**결과**: 깔끔. Expo autolink가 podspec 자동 처리. JS 측은 `requireNativeModule`로
타입 있는 브리지.

## ADR-007 — 화면 컴포넌트 분리 기준 (2026-04)

**문맥**: `CryMonitorScreen.tsx` 가 595줄로 비대화. 다음 작업(히스토리)에서 일부
컴포넌트 재사용 필요.

**결정**:
- 한 화면 300줄 넘고, 같은 패턴이 다른 화면에서 재사용될 가능성 있으면 분리
- 분리 위치: `src/components/<도메인>/<Component>.tsx`
- 순수 헬퍼는 `src/utils/`

**결과**: 595 → 287줄. ResultView/AcousticDetails/LearningStageBanner는 향후
히스토리 디테일 페이지에서 재사용 가능.

## ADR-008 — 인증 미도입 (2026-04, 잠정)

**문맥**: 가족 초대 코드 8자리만으로 데이터 접근.

**리스크**: 코드 추측당하면 모든 기록 조회/수정 가능.

**현재 상태**: 인증 안 만듦. 1차 출시 (지인 + 가족) 단계에선 OK 판단.

**트리거**: 외부 사용자 100명 넘거나, 데이터 유출 사고 1건이라도 발생 시 즉시 도입.

자세한 건 [`운영/보안.md`](../운영/보안.md).

5. **결정** + 굵게
6. **근거**
